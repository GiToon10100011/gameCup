## 1. 문서 개요

GameCup PRD를 기반으로 사용자 목표 중심의 유즈케이스를 정의한다.  
각 유즈케이스는 액터(Actor), 사전조건, 기본 흐름, 예외·대안 흐름, 사후조건을 포함한다.  
UC-01~UC-04는 PRD Iteration 2/3 기반이며, UC-05~UC-10은 PRD Iteration 4 (v4.0) 기반으로 추가되었다.

---

## 2. 액터(Actor) 정의

|액터|설명|
|---|---|
|**일반 사용자 (Guest)**|비로그인 상태로 서비스를 이용하는 모든 방문자. MVP 범위의 주요 액터.|
|**인증 사용자**|이메일 매직 링크로 로그인하여 토너먼트 저장·이력·공유 기능을 이용하는 사용자. Iteration 4 주요 액터.|
|**비인증 방문자 (공유 링크)**|공개 결과 링크(F-20)를 통해 결과를 열람하는 비로그인 방문자.|
|**외부 게임 API**|게임 데이터(이름, 썸네일 등)를 제공하는 외부 시스템 액터 (RAWG API).|
|**Supabase Auth**|매직 링크 이메일 발송 및 인증 세션 관리를 담당하는 외부 인증 시스템 액터.|

## 4. 유즈케이스 상세 명세

### UC-01 · 게임 검색하기

|항목|내용|
|---|---|
|**액터**|일반 사용자, 외부 게임 API|
|**목표**|사용자가 원하는 게임을 이름으로 검색하여 후보에 추가하기 위한 정보를 얻는다.|
|**사전조건**|사용자가 서비스에 접근한 상태이다.|
|**기본 흐름**|1. 사용자가 검색창에 게임 이름을 입력한다.<br>2. 시스템은 외부 게임 API에 검색 요청을 전송한다.<br>3. 외부 API가 결과를 반환하면, 시스템은 게임 썸네일과 이름을 목록으로 표시한다.<br>4. 사용자는 결과 목록을 확인한다.|
|**예외·대안 흐름**|**2a.** 동일 검색어에 대한 이전 결과가 캐시에 존재하면, API 호출 없이 캐시 결과를 표시한다. `(NF-05)`<br>**1a.** 검색어가 비어 있거나 공백만 입력된 경우, API를 호출하지 않고 목록을 표시하지 않는다. `(F-13)`<br>**3a.** 외부 API 호출이 실패한 경우, 오류 메시지를 표시하고 이전 상태를 유지한다. `(F-12)`|
|**사후조건**|검색 결과 목록이 화면에 표시된 상태이다.|
|**관련 요구사항**|F-01, F-02, F-12, F-13, NF-01, NF-05|

### UC-02 · 토너먼트 후보 구성하기

|항목|내용|
|---|---|
|**액터**|일반 사용자|
|**목표**|사용자가 1:1 대결에 참가할 게임 후보 목록을 자유롭게 구성한다.|
|**사전조건**|UC-01이 완료되어 검색 결과가 표시된 상태이다.|
|**기본 흐름**|1. 사용자가 검색 결과 목록에서 원하는 게임을 선택한다.<br>2. 시스템은 선택된 게임을 후보 목록에 추가한다.<br>3. 사용자는 추가·제거를 반복하며 후보 목록을 원하는 구성으로 만든다.<br>4. 사용자가 후보 목록을 최종 확인한다.|
|**예외·대안 흐름**|**2a.** 이미 후보 목록에 있는 게임을 추가하려 하면, 중복 안내 메시지를 표시하고 추가하지 않는다. `(F-04)`<br>**3a.** 사용자가 후보 목록에서 특정 게임을 삭제하면, 해당 게임이 목록에서 제거된다. `(F-05)`|
|**사후조건**|후보 목록에 1개 이상의 게임이 등록된 상태이다.|
|**관련 요구사항**|F-03, F-04, F-05|

### UC-03 · 토너먼트 진행하기

|항목|내용|
|---|---|
|**액터**|일반 사용자|
|**목표**|사용자가 1:1 대결 방식으로 게임들을 비교하여 최종 1개를 가린다.|
|**사전조건**|UC-02 완료 후 후보 목록에 2개 이상의 게임이 등록되어 있다.|
|**기본 흐름**|1. 사용자가 토너먼트 시작을 요청한다.<br>2. 시스템이 후보 게임을 무작위 순서로 배치하여 첫 번째 라운드를 구성한다.<br>3. 시스템은 1:1 대결 화면을 표시한다.<br>4. 사용자가 두 게임 중 선호하는 게임을 선택한다.<br>5. 선택된 게임이 다음 라운드 진출 목록에 등록된다.<br>6. 한 라운드의 모든 대결이 완료되면, 시스템이 자동으로 다음 라운드를 구성한다.<br>7. 게임이 1개 남을 때까지 3~6을 반복한다.|
|**예외·대안 흐름**|**1a.** 후보가 2개 미만이면 시작 버튼이 비활성화되며 토너먼트를 시작할 수 없다. `(F-06)`<br>**2a.** 후보 게임이 홀수 개이면 마지막 게임은 부전승으로 자동 다음 라운드 진출한다. `(F-10)`|
|**사후조건**|모든 라운드가 종료되어 최종 1개의 게임이 결정된 상태이다.|
|**관련 요구사항**|F-06, F-07, F-08, F-09, F-10, NF-02|

### UC-05 · 매직 링크 로그인

|항목|내용|
|---|---|
|**액터**|인증 사용자, Supabase Auth|
|**목표**|사용자가 이메일 매직 링크를 통해 로그인하여 인증 세션을 수립하고 보호 리소스에 접근한다.|
|**사전조건**|사용자가 서비스에 접근한 상태이다. 세션이 존재하지 않거나 사용자가 로그인을 요청한 상태이다.|
|**기본 흐름**|1. 사용자가 로그인 화면(AuthPage)에서 이메일 주소를 입력하고 "링크 전송"을 클릭한다.<br>2. 시스템(AuthModule)이 SupabaseClient를 통해 Supabase Auth에 매직 링크 발송을 요청한다.<br>3. Supabase Auth가 해당 이메일로 링크가 포함된 메일을 발송한다.<br>4. 시스템이 "이메일을 확인하세요" 안내 메시지를 표시한다.<br>5. 사용자가 이메일에서 링크를 클릭하면 앱의 콜백 URL로 진입한다.<br>6. 시스템이 onAuthStateChange 이벤트를 수신하고 세션(IUser)을 AuthSlice에 저장한다.<br>7. 시스템이 인증 완료 처리 후 토너먼트 허브(HubPage)로 리다이렉트한다.|
|**예외·대안 흐름**|**1a.** 유효하지 않은 이메일 형식을 입력한 경우, 시스템은 입력 오류를 안내하고 발송을 차단한다. `(F-14)`<br>**5a.** 링크가 만료되었거나 콜백 처리 중 오류가 발생한 경우, 시스템은 오류 메시지를 표시하고 이메일 재입력 화면으로 돌아간다. `(F-14)`<br>**5b.** 사용자가 메일을 수신하지 못한 경우, 재발송 버튼을 통해 2단계부터 재시도할 수 있다. `(F-14)`<br>**[인증 가드]** 비로그인 사용자가 보호 리소스(내 토너먼트 목록·저장·이력)에 접근하면, 시스템이 접근을 차단하고 AuthPage로 리다이렉트한다. `(F-15)`|
|**사후조건**|사용자의 인증 세션이 수립되어 AuthSlice에 IUser가 저장된 상태이다. 사용자가 HubPage에 위치한다.|
|**관련 요구사항**|F-14, F-15, NF-07|

### UC-06 · 토너먼트(설문) 생성·저장

|항목|내용|
|---|---|
|**액터**|인증 사용자|
|**목표**|로그인 사용자가 게임 검색과 후보 등록을 거쳐 후보 세트에 이름을 붙여 토너먼트(설문)로 저장한다.|
|**사전조건**|UC-05가 완료되어 인증 세션이 수립된 상태이다. 사용자가 토너먼트 생성 화면(CreatePage)에 진입한 상태이다.|
|**기본 흐름**|1. 사용자가 게임 이름을 검색창에 입력하여 게임을 검색한다. (UC-01 재사용)<br>2. 사용자가 검색 결과에서 원하는 게임을 선택하여 후보 목록에 추가한다. (UC-02 재사용)<br>3. 사용자가 원하는 구성이 될 때까지 1~2를 반복한다.<br>4. 후보가 2개 이상 등록되면 토너먼트 이름 입력 필드가 활성화된다.<br>5. 사용자가 토너먼트 이름을 입력하고 "저장" 버튼을 클릭한다.<br>6. 시스템(TournamentStorageModule)이 CandidateModule을 통해 후보 세트를 수집하고 SupabaseClient를 통해 Supabase에 저장한다.<br>7. 저장이 완료되면 시스템이 TournamentLibrarySlice에 활성 토너먼트를 설정하고 HubPage로 이동한다.|
|**예외·대안 흐름**|**4a.** 후보가 2개 미만인 경우, 저장 버튼이 비활성화되며 저장할 수 없다. `(F-16)`<br>**5a.** 토너먼트 이름이 비어 있는 경우, 시스템은 이름 입력을 요구하고 저장을 차단한다. `(F-16)`<br>**6a.** Supabase 저장 중 오류가 발생하면, 시스템은 오류 메시지를 표시하고 저장 이전 상태를 유지한다. `(F-16)`<br>**[미인증]** 인증 세션이 없는 상태에서 저장을 시도하면, 시스템이 AuthPage로 리다이렉트한다. `(F-15)`|
|**사후조건**|토너먼트가 Supabase에 사용자별로 저장되어 있다. 사용자가 HubPage 토너먼트 목록에서 해당 토너먼트를 확인할 수 있는 상태이다.|
|**관련 요구사항**|F-16, F-01~F-05 (재사용), F-15, NF-06|

### UC-07 · 내 토너먼트 선택·플레이 진입

|항목|내용|
|---|---|
|**액터**|인증 사용자|
|**목표**|로그인 사용자가 허브 화면의 내 토너먼트 목록에서 하나를 선택하여 플레이(토너먼트 진행)를 시작하거나, 필요 시 삭제한다.|
|**사전조건**|UC-05가 완료되어 인증 세션이 수립된 상태이다. Supabase에 저장된 토너먼트가 1개 이상 존재한다. 사용자가 HubPage에 위치한 상태이다.|
|**기본 흐름**|1. 시스템이 TournamentStorageModule을 통해 SupabaseClient에서 사용자의 토너먼트 목록을 불러와 허브 화면에 표시한다.<br>2. 사용자가 목록에서 원하는 토너먼트를 선택한다.<br>3. 시스템이 선택된 토너먼트의 후보 세트를 불러오고 TournamentLibrarySlice에 활성 토너먼트를 설정한다.<br>4. 시스템이 PlayPage로 이동하여 토너먼트 진행을 시작한다. (UC-03으로 이어짐)|
|**예외·대안 흐름**|**1a.** 목록 조회 중 네트워크 오류가 발생하면, 시스템은 오류 메시지를 표시하고 재시도 버튼을 제공한다. `(F-17)`<br>**[삭제]** 사용자가 토너먼트 옆 삭제 버튼을 클릭하면, 시스템이 확인 요청 후 TournamentStorageModule.deleteTournament를 호출하여 해당 토너먼트를 삭제하고 목록을 갱신한다. `(F-17)`<br>**[빈 목록]** 저장된 토너먼트가 0개이면 허브는 온보딩 상태(UC-08)로 분기한다. `(F-17, F-18)`|
|**사후조건**|선택된 토너먼트가 TournamentLibrarySlice의 activeTournament로 설정되어 있다. PlayPage에서 토너먼트가 진행 중인 상태이다.|
|**관련 요구사항**|F-17, F-15, NF-06|

### UC-08 · 온보딩 — 첫 토너먼트 생성

|항목|내용|
|---|---|
|**액터**|인증 사용자|
|**목표**|로그인 후 저장된 토너먼트가 없는 사용자에게 첫 토너먼트 생성을 단계적으로 안내하여 서비스에 자연스럽게 진입시킨다.|

### UC-10 · 결과 공유(링크)

|**사후조건**|[공유 생성] IPublicShare가 Supabase에 저장되어 있고, 공개 URL이 사용자에게 제공된 상태이다.<br>[링크 열람] 비인증 방문자가 PublicResultPage에서 결과를 확인한 상태이다.|
