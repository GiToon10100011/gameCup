## 1. 문서 개요

PRD v3.0(Iteration 3) 기반으로, Iteration 3 §5 "범위 외 항목"에서 예고했던 결과 서버 저장·공유 및 사용자 인증을 정식 요구사항으로 확정하고, 서비스 구조를 **단일 세션 토너먼트**에서 **멀티 토너먼트 허브** 모델로 전환하여 갱신한 문서이다.

| 항목 | 내용 |
| --- | --- |
| **목표** | 사용자가 복수의 토너먼트(설문)를 저장·반복 플레이하고, 결과를 이력으로 누적·공유하는 인터랙티브 게임 취향 선별 서비스 구축 |
| **핵심 가치** | 빠른 게임 검색 경험 + 토너먼트 방식의 직관적 취향 선별 + 내 취향 이력 보존 + 공유 |
| **주요 변경** | 사용자 인증(Supabase Auth 매직 링크) 도입 / 메인 화면을 토너먼트 허브로 재편 / 토너먼트(설문) 저장·관리 / 결과 이력 누적 / 공개 URL 기반 결과 공유 / 첫 사용자 온보딩 시퀀스 |

> 서비스의 멀티 토너먼트·인증 플로우를 구성하는 핵심 신규 기능. Iteration 4 내 반드시 구현되어야 한다.

| --- | --- | --- | --- |

## 4. 비기능 요구사항

> 시스템의 품질 속성을 정의한다. 특정 구현 방법이 아닌, 시스템이 만족해야 하는 조건을 기술한다.

| ID | 분류 | 요구사항 | 변경 사유 |
| --- | --- | --- | --- |
| NF-01 | **응답성** | 검색 입력 후 1초 이내에 결과 목록이 화면에 표시되어야 한다. | - |
| NF-02 | **안정성** | 짧은 시간 내 연속 클릭 등 비정상적인 입력이 발생하더라도 데이터 불일치 없이 정상 상태를 유지해야 한다. | - |
| NF-03 | **브라우저 호환성** | Chrome, Edge, Firefox의 최신 안정 버전에서 동일하게 동작해야 한다. | - |
| NF-04 | **확장성** | 향후 기능 추가를 위해 상태 관리 및 화면 구조가 기능 단위로 분리되어야 한다. | - |
| NF-05 | **불필요한 외부 호출 최소화** | 동일한 검색어에 대한 외부 API 응답은 세션 내에서 재활용되어야 하며, 중복 호출이 발생하지 않아야 한다. | - |
| NF-06 | **데이터 보안·격리** | Supabase Row Level Security(RLS)를 적용하여, 사용자의 토너먼트·결과 이력 데이터는 본인 계정으로만 접근 가능해야 한다. 공개 링크로 공유된 결과는 예외적으로 비인증 읽기를 허용한다. | 신규 추가 (EXP-01) |
| NF-07 | **인증 세션 신뢰성** | 매직 링크 세션은 Supabase Auth의 표준 토큰 갱신(refresh) 메커니즘을 통해 유지되어야 하며, 네트워크 일시 단절 후 복구 시에도 세션이 자동으로 재연결되어야 한다. | 신규 추가 (EXP-01) |

## 6. 아키텍처·화면 구조 변경 요지

> 상세 클래스 다이어그램·시퀀스 다이어그램은 `docs/03-design/uml-v2.0.md`에서 별도로 다룬다.

### 6.1 메인 화면 재편

- **Iteration 3:** 메인 화면 = "게임 검색 입력 + 후보 목록"(바로 후보 선택 진입)
- **Iteration 4:** 메인 화면 = **토너먼트 허브** — 내 토너먼트 목록(F-17) 또는 첫 사용자 온보딩(F-18). "새 토너먼트 만들기" 버튼으로 게임 검색·후보 등록 화면(F-01~F-06)에 진입한다.

### 6.2 기존 컴포넌트 재사용 방침

Iteration 3(Epic #1) 산출물인 `SearchInput`, `SearchDropdown`, `CandidateList`, `DuplicateToast`, `ErrorMessage` 컴포넌트와 `searchModule`, `candidateModule`은 **폐기하지 않는다.** 이들은 "토너먼트 생성/편집" 플로우에서 그대로 재사용된다.

### 6.3 Data 계층 변경

- **추가:** Supabase 클라이언트(`supabaseClient`)가 Data 계층에 합류한다.
- **기존:** RAWG API Gateway(`externalApiClient`) + Zustand StateStore 유지.
- Supabase는 인증 세션 관리, 토너먼트(설문) CRUD, 결과 이력 저장·조회를 담당한다.

### 6.4 3계층 계층 호출 방향

기존 Presentation → Business → Data 방향을 그대로 유지한다.  
신규 Supabase 관련 호출도 Business 레이어(신규 모듈: `authModule`, `tournamentStorageModule` 등)를 경유해야 하며, Presentation이 `supabaseClient`를 직접 호출하는 것은 금지한다.
